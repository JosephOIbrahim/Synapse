# SYNAPSE Latency × Solaris Convergence — First-Principles Report

## 1. THESIS

**Confirmed, with one sharpening.** Latency reduction (A) and Solaris one-shot building (B) converge on a single lever: **route a multi-node Solaris build to ONE declarative call instead of an imperative per-node loop.** The first-principles model says the dominant cost is the *number of sequential assistant turns* plus the *~2s main-thread floor × number of cooks*. A from-scratch render-ready Karma scene is ~7 nodes + ~6 wires + several encoded-parm sets ≈ 13–30 imperative tool calls today; the post-mortem hit the 25-iteration cap *without finishing*. One `synapse_solaris_build_graph` (template) call collapses that to **1 turn, 1 main-thread hop, 1 terminal cook** — and depth (engine, camera, AOVs) becomes free because the handler materializes the whole composed DAG inside one `hou.undos.group` (`handlers_solaris_graph.py:330-416`). Same edit serves both axes.

**The sharpening — the lever is currently un-pullable for three independent reasons, all verified:**

1. **The richest one-shot builder is invisible.** `build_karma_xpu_shot()` (engine=xpu + camera-bind + resolution + productName, `solaris_compose_tools.py:92,172-175`) is registered as a `/synapse` command (`handlers.py:551-553`) but has **zero** entries in `_tool_registry.TOOL_DEFS` (grep count = 0). The model literally cannot see the tool purpose-built for the GOAL request.
2. **The reachable declarative tool is steered against and schema-trapped.** The interactive panel *does* advertise `synapse_solaris_build_graph` (full toolset, `synapse_panel.py:1272-1280`, `enforce_worker_policy=False`), but the system prompt tells the model to *"Chain multiple tool calls"* (`system_prompt.py:74`), leads with a per-node Chain-Insertion pattern (`:123-131`), and its schema marks `required:["nodes","connections"]` (`_tool_registry.py:706`) even though the handler fully expands a template from empty inputs (`handlers_solaris_graph.py:217-235`). So a template-only call *looks* schema-invalid → the model falls back to imperative.
3. **On the autonomous path the collapse is structurally impossible.** `build_graph`/`assemble_chain` are non-read-only and map to `build_from_manifest=review`; `worker_policy` standard mode (the default) denies every review/approve/critical tool and fail-closes unknown tools (`worker_policy.py:48,56,120-141`). The autonomous worker therefore never sees build_graph, never sees the compose tools, and `execute_python` is `critical` → it is *forced* into the slowest inform-gated `create/connect/set` loop — the exact mode that hit the cap.

**Net:** the convergence thesis holds, but "make it the default" requires unblocking all three. The good news: every fix is small and most are Houdini-free.

## A. PHANTOM THAT SLIPPED INTO THE REPORT (§5, prompt-caching rejection — WRONG)

The report rejected Anthropic prompt-caching as "shaves prefill ms under the ~2s floor — phantom." **That reasoning is false here, and I measured it:**

- `anthropic_provider.py:87-95` builds the request body with `tools` and `system` and **zero `cache_control`** anywhere. Caching is OFF.
- The advertised tool block is **65,699 chars ≈ 18,771 tokens** (118 tools, measured live via `tool_bridge.get_anthropic_tools()`), re-sent and **re-prefilled on every one of the 25 turns**.

19k static tokens re-prefilled 25× is **~475k redundant prefill tokens**, not "ms." Prefill latency compounds with turn count — the *exact* dominant term the report privileges. Caching turns 24 of those 25 prefills into cache-reads. It's Houdini-free, ~5 lines (cache_control breakpoints on the tools array + system), and — unlike the 25→1 collapse — it *also* speeds every conversation that legitimately can't collapse (debugging, iteration, Q&A). The report's §5 conflated "small prefix" with a 19k-token prefix. **This is the phantom-rejection error.** (Priority: below the turn-collapse, above most U-items.)
