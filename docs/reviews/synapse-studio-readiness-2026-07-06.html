Part I — What shipped

The context-capability track — a per-context review that improves what it measures

Your request was a harness that reviews SYNAPSE's ability to affect Houdini per context, and on that same per-context basis grinds SYNAPSE's ability to create — using dynamic workflows and agent teams. Rather than build a second loop, this grafts onto the existing harness/run.ts as a third state-file trigger, peer of the H22 drop and the v6 blueprint drop. A spec was frozen, then four builder agents implemented it against disjoint file sets on the live tree; an adversarial pass and the full suite verified it.

The loop
C.0Probe all six contexts by driving SynapseHandler.handle() — the same seam the live WebSocket transport calls, never raw hou for mutations — then deposit the capability catalog + a review sweep.
↓Committing the catalog arms C.1–C.6: one improvement sprint per context (SOP · LOP · COP · TOP · DOP · MAT).
C.nEach sprint is gated by its context's golden (create → verify → revert) and a ratchet — the gap count must strictly decrease, or already be zero — proven against the HEAD baseline so a sprint can't grade its own refresh.
Governance held intact
✓Merge stays the human gate — the harness commits in worktrees only, never to master.
✓New cycle class queued ratified:false in the flywheel queue — the same human sign-off U.5 got, per the anti-runaway anchor.
✓Renders excluded from every golden (Indie husk no-ops silently); APEX/rigging excluded structurally by the no_rigging_drift guardrail. Existing guardrails run on every C-sprint for free.

Part III — Deployment review

24 findings, ranked, tagged by the contexts they bite

Every finding below was cross-checked by an adversarial verifier whose default verdict was refuted; none were. CONFIRMED means reproduced from code; PARTIAL means the core holds but a stated detail or severity was corrected — those corrections are called out, because several lowered the risk. The severity shown is the post-verification rating. The solo-localhost posture is deliberate and test-pinned; findings say precisely which contexts each gap actually harms.

criticalconfirmed studiofarmsolo

Consent gates auto-approve or are absent on every live execution path

The one production bridge is the panel singleton, built with an unconditional-true callback and then explicitly disarmed (_bridge._gate = None). The /mcp dispatch shares that same disarmed singleton; the live /synapse transport bypasses the bridge entirely. So REVIEW / APPROVE / CRITICAL gates on execute_python, delete_node, render submits and exports fire nowhere at runtime, while the IntegrityBlock records consent_verified = true. HumanGate.propose has zero live producers — the panel's consent auto-surface consumes proposals that never arrive.

note Solo-localhost auto-approve is documented and forced by a real GUI-thread deadlock — but the bridge-path disarm contradicts the repo's own Safety Rule 5 even for solo.

panel/bridge_adapter.py:202 · mcp/tools.py:118 · server/handlers.py:1051 · shared/bridge.py:762

criticalconfirmed studiotd

Studio auth/RBAC fails open on the shared-key path and is absent on the production transport

check_permission() runs only in websocket.py — the non-default testing backend. The production hwebserver /synapse and /mcp transports have shared-key auth but no roles, RBAC, or per-user sessions. Even on websocket, RBAC is guarded by if user_session: with no else: a shared-key client with no directory entry authenticates, gets no session, skips permission checks, and lands in the audit log with user_id = ''. This is tracked internally as SEC-1.

corrected The /mcp fail-open is except ImportError: pass (fails open only if the auth module can't load), not a bare except.

server/websocket.py:591,447 · server/hwebserver_adapter.py:232 · mcp/server.py:690

criticalconfirmed studio

execute_python is unsandboxed full-__builtins__ RCE, LAN-reachable in studio mode

The live handler executes arbitrary Python with full builtins and hou in scope, no import filter, no length cap, no gate — deliberately pinned ungated for single-user localhost. In studio mode the same handler is reachable over the network by anyone holding the shared key (or, per the finding above, no key at all), turning a create tool into remote code execution on every artist's workstation.

server/handlers.py:1021–1097 (pinned by tests/test_phase0b_consent_posture.py)

criticalpartial studio

One shared handler holds per-connection identity in mutable state → cross-user audit misattribution

A single SynapseHandler instance carries _user_id as mutable state that each connection overwrites via set_user_id. Under concurrent studio clients, one user's command can be stamped with another's identity in the audit trail — the accountability the studio posture exists to provide.

corrected Not every on-disk decision is anonymous — the gate store does persist decided_by=user_id. The race is on the hash-chained audit JSONL, which is the narrower but still real claim.

server/handlers.py:301,342,456

High — architecture & policy root
highconfirmed studiotdfarmsolo

No single source of truth for policy: five disagreeing taxonomies, default-open on the bridge path

Capability, gate level, read-only status, and disk-touch are each defined more than once across the bridge, the handlers, the tool registry, and the worker policy — and they disagree. The bridge path defaults open when a lookup misses. This is the architectural root the four criticals grow from: with no one authoritative policy table, there is nowhere to enforce consent or permission consistently.

shared/bridge.py · mcp/_tool_registry.py · server/handlers.py · panel/worker_policy.py

highconfirmed tdfarmstudio

Scout's phantom-API gate mis-serves on build mismatch; one point-build table can't serve a mixed fleet

The phantom-API defense keys on a single introspected symbol table for one exact Houdini build. In the external-process topology the version check is skipped, and a studio running several H21 point-builds (or H21 + H22) has one table that is authoritative for at most one of them — the gate silently over- or under-serves the rest. Directly relevant to the new context track, whose catalog is likewise per-build.

cognitive/tools/scout.py · cognitive/tools/data/h21_symbol_table.json

highconfirmed solostudiofarmtd

Fake-hou module residency is systemic and unguarded; suite validity depends on collection order

Dozens of test files plant fake hou modules in sys.modules at import time; the alphabetically-first planter wins for the whole run. ~4000 tests are green in standalone mode but never touch a real Houdini session, and nothing guards against a residency collision quietly changing what the suite actually exercises. A TD cannot fully trust the suite as an upgrade gate.

High — memory & provenance
highconfirmed studiosolotd

Memory provenance is mislabeled at write time — AI decisions stamped source='user'

Model-authored memories are written with source = 'user', markdown entries carry no author, and the trust-tier fields are dead. The provenance discipline the docs describe — knowing whether a remembered fact came from the artist or the model — does not survive the write path, so low-trust model inferences are indistinguishable from operator intent.

memory/ write path · tier fields unused

highconfirmed solostudiofarm

Recall has no staleness, conflict, or recency semantics — stale memories can outrank fresh facts

Retrieval has no notion of recency or contradiction and its order is nondeterministic, so an outdated memory can outrank a newer, conflicting scene fact. Context assembly is treated as a ranking problem, not a trust problem.

synapse_recall / synapse_memory_query handlers

highconfirmed studiofarmtd

Three parallel memory stores, per-process env-selected — split-brain by construction

JSONL, Living-Memory markdown, and Moneta all exist, with the active backend chosen by an environment variable per process. Two artists — or a panel and a farm job — with different env settings write to different stores that can silently diverge, with no reconciliation authority.

Part IV — Prioritized upgrade path

Silent safety first, then durability, then the eval wedge

Ordered by the evaluation playbook's priority: silent safety gaps, then duplicate-write / durability, then evaluation coverage, then farm correctness. Step 1 alone collapses four of the five safety findings.

1

Unify the policy layer and enforce consent at the dispatch boundary

Collapse the five disagreeing taxonomies into one authoritative table keyed by gate level, and check it inside registry.invoke() / handle() so every transport hits the same policy. Give /mcp and autonomous paths their own HumanGate-wired bridge instead of the disarmed panel singleton; make auto-approve an explicit per-deploy-mode choice; record the real consent source in the IntegrityBlock; default-deny in studio mode.

closes criticals 1 · 3 · 4 + the policy-root high — safety-permissions
2

Carry per-user identity through dispatch

Move authn/RBAC into the transport-agnostic boundary keyed by per-connection identity; require per-user keys on all three transports; end the shared-handler mutable _user_id race so audit entries attribute correctly; default-deny when role/session is unresolved.

closes critical 2 + the identity race — safety-permissions
3

Choose one authoritative memory store and make provenance real

Pick a single source of truth among JSONL / Living-Memory / Moneta with explicit reconciliation; stamp source, author, and trust tier at write time; add recency and conflict handling to recall so a stale memory can never outrank a fresher, contradicting scene fact.

closes the three memory highs — context-memory-eval / architecture
4

Turn the context track into the evaluation backbone

The C-track goldens are the first automated create→verify→revert demonstrations — extend them to a headless/farm scenario, and swap the harness's size>1024 stub for the real validate_frame that already ships. Guard the fake-hou residency so suite validity stops depending on collection order, and pin permission boundaries as invariants, not enumerated regressions.

closes the golden-stub, render-stub, and fake-hou highs — context-memory-eval
5

Close the farm-headless creation gaps the catalog just surfaced

The TOP hdefereval GUI-binding and the DOP sim-population failure are now the C.4 / C.5 targets — make TOP tools work headless (or declare them GUI-only, honestly) and give DOP a create path that populates the simulation. Neutralize the latent PDG remove_files=True rollback before any caller reaches it, and per-build the scout symbol table so a mixed fleet is served correctly.

C.4 · C.5 improvement sprints + the PDG and scout highs — farm / td
